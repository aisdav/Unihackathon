1. ЧТО МЫ СДЕЛАЛИ (в одном абзаце, для вступления)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Мы создали AI-платформу, которая автоматически проверяет технические
задания научных проектов. Учёный или администратор загружает документ,
система анализирует его по пяти критериям, выставляет оценку от 0 до 100,
показывает конкретные замечания с объяснением, предлагает рекомендации,
генерирует улучшенную версию текста, отдаёт PDF-отчёт и отвечает на вопросы
через AI-чат. Дополнительно реализован «Простой режим» для учёных с низкой
цифровой уверенностью или пожилого возраста.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
2. ПРОБЛЕМА, КОТОРУЮ МЫ РЕШАЕМ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Большинство ТЗ научных проектов страдают от:
- Расплывчатых формулировок («будет разработан современный подход»)
- Отсутствия измеримых критериев успеха (KPI)
- Пропущенных обязательных разделов
- Логических противоречий между целью, задачами, сроками и бюджетом

Экспертная проверка вручную занимает дни. Мы делаем это за секунды.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
3. СТЕК И АРХИТЕКТУРА
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Frontend:  React 18 + Vite + Tailwind CSS + Nginx
Backend:   FastAPI (Python) + SQLAlchemy + Alembic
AI:        OpenAI GPT + ChromaDB (векторная БД для RAG) + pymupdf4llm
Data:      PostgreSQL + Redis
Infra:     Docker Compose (5 контейнеров: frontend, backend, db, redis, celery_worker)

Поток данных:
  Пользователь → Nginx → FastAPI → Document Parser
                                 → AI Pipeline (5 агентов)
                                 → PostgreSQL / ChromaDB
                                 → Отчёт / Чат / API


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
4. AI PIPELINE — 5 АГЕНТОВ (главное, что нужно уметь объяснить)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Шаг 1 — StructureAgent
  Проверяет наличие обязательных разделов: цель, задачи, сроки,
  ожидаемые результаты, бюджет, критерии успеха.
  Выдаёт список «пропущенных секций».

Шаг 2 — RAG Lookup (Retrieval-Augmented Generation)
  Сравнивает документ с базой хороших и плохих примеров ТЗ
  в ChromaDB. Достаёт похожие фрагменты и передаёт их в следующий шаг
  как контекст. Это значит — модель судит не в вакууме, а опираясь
  на реальные прецеденты.

Шаг 3 — QualityAgent
  Анализирует текст на размытые формулировки («Anti-vague detector»),
  неконкретные обещания, отсутствие количественных показателей.
  Генерирует список замечаний с severity (critical / warning / info).

Шаг 4 — ConsistencyAgent
  Проверяет логическую связность: соответствуют ли задачи цели,
  реалистичны ли сроки, совпадает ли бюджет с объёмом работ,
  есть ли противоречия внутри документа.

Шаг 5 — Improved Text Generation
  На основе всех выявленных проблем генерирует улучшенную версию
  документа и пример правильного ТЗ как образец.

Итог: оценка 0-100 по 5 критериям (20 баллов каждый):
  • Структура    — есть ли все разделы
  • Ясность      — конкретность формулировок
  • Полнота      — достаточно деталей для работы
  • Измеримость  — KPI и критерии успеха
  • Логика       — цель / задачи / сроки / ресурсы согласованы


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
5. КЛЮЧЕВЫЕ ФИЧИ ДЛЯ ПОКАЗА НА ДЕМО
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Score Gauge — круговой индикатор с цветовой кодировкой (красный/жёлтый/зелёный)
2. Score Breakdown — 5 полос с оценкой по каждому критерию
3. Issues — список замечаний по severity, с раскрытием подробностей
4. Recommendations — конкретные рекомендации по исправлению
5. Section Checklist — галочки по обязательным разделам
6. Logic Graph — схема логической цепочки документа
7. Improved Text — готовый улучшенный текст ТЗ
8. PDF Report — скачиваемый отчёт
9. AI Chat (streaming) — вопросы по документу в реальном времени
10. Простой режим — упрощённый UI для пожилых пользователей

6. СЦЕНАРИЙ ДЕМО (порядок показа)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Открыть http://localhost:3000
2. Зарегистрироваться или войти
3. Загрузить тестовый документ (плохое ТЗ — из backend/tests/fixtures/bad_tz_example.txt)
4. Нажать «Запустить анализ» → подождать статус «Готово»
5. Показать Score Gauge с низкой оценкой (будет красный)
6. Раскрыть несколько замечаний — показать конкретные цитаты из текста
7. Перейти на вкладку «Что исправить» — показать рекомендации
8. Перейти на «Готовый текст» — показать улучшенную версию
9. Скачать PDF-отчёт
10. Задать вопрос в чате: «Какой главный недостаток этого ТЗ?»
11. Включить «Простой режим» — показать другой UI
12. Загрузить хорошее ТЗ — показать высокую оценку для контраста

7. ОТВЕТЫ НА ЧАСТЫЕ ВОПРОСЫ ЖЮРИ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

В: Почему не просто ChatGPT?
О: ChatGPT — общая модель без специфики научных ТЗ. Мы добавили:
   RAG по базе реальных примеров, структурированный multi-step pipeline,
   специализированные агенты, хранение истории и документов, PDF-отчёты.
   Это не чат-бот, это продукт с конкретным UX-флоу.

В: Как вы оцениваете качество 0-100?
О: Пять агентов дают оценку по 20 баллов каждый. Итог — взвешенная сумма.
   Каждый критерий проверяется отдельным prompt'ом с чёткими инструкциями,
   что считать 0, что 10, что 20 баллов.

В: Что такое RAG?
О: Retrieval-Augmented Generation. Перед тем как анализировать документ,
   система ищет в ChromaDB похожие примеры хороших и плохих ТЗ.
   Эти примеры подставляются в контекст модели — она судит не абстрактно,
   а сравнивая с реальными прецедентами. Это снижает галлюцинации.

В: Это реально работает или только имитация?
О: Работает. Можно прямо сейчас загрузить файл и получить результат.
   API задокументировано на /docs. Все контейнеры подняты.

В: Почему Celery если он не используется?
О: Celery заложен в архитектуру под масштабирование. Сейчас анализ идёт
   через FastAPI background tasks — это достаточно для демо.
   Переход на Celery tasks — следующий шаг после хакатона.

В: Как обеспечивается безопасность?
О: JWT-аутентификация, пароли хэшируются через pbkdf2_sha256,
   каждый пользователь видит только свои документы.
   .env с ключами не попадает в репозиторий.

В: Как масштабировать?
О: Горизонтально через Docker Swarm или Kubernetes,
   Celery workers для параллельного анализа,
   ChromaDB можно заменить на облачный Pinecone/Weaviate,
   PostgreSQL — на managed cloud DB.

В: Почему «Простой режим»?
О: Целевая аудитория — учёные. Многие из них в возрасте 50+ и не привыкли
   к сложным UI. Простой режим убирает визуальный шум, увеличивает шрифты,
   усиливает контраст, даёт понятные подписи вместо технических терминов.
   Это реальная accessibility-фича, а не косметика.

В: Какие форматы поддерживаете?
О: PDF, DOCX, TXT. Лимит — 10 MB на файл.

В: Где хранятся документы?
О: На сервере в файловой системе, метаданные — в PostgreSQL,
   векторные эмбеддинги — в ChromaDB.

8. КАК ПРИВЯЗАТЬ К КРИТЕРИЯМ ХАКАТОНА
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Стратегическая релевантность:
  → Цифровизация управления наукой. Государство вкладывает миллиарды
    в научные проекты — качество ТЗ напрямую влияет на результат.
    Автоматизация экспертизы снижает стоимость и ускоряет цикл.

Научная новизна:
  → Multi-agent RAG pipeline специально для научных ТЗ.
    Не переобученная модель, а архитектурное решение из нескольких
    специализированных агентов с общей памятью.

Практическая применимость:
  → Готовый продукт, запускается одной командой docker compose up.
    Конечный пользователь — учёный, которому не нужно знать про AI.
    Простой режим снижает барьер входа для немолодых специалистов.

Ожидаемые результаты:
  → Снижение времени экспертизы ТЗ с нескольких дней до минут.
    Повышение качества документов на входе в проект.
    Снижение числа доработок и согласований.

Соц-экономический эффект:
  → Экономия экспертного времени в НИИ и университетах.
    Доступность инструмента для учёных любого возраста.
    Снижение риска провала проектов из-за плохо написанных ТЗ.

Реализуемость:
  → Проект уже работает. Не макет, не слайды — живой прототип.
    Стек стандартный, команда поддерживает его самостоятельно.

9. ТЕХНИЧЕСКИЕ ДЕТАЛИ, КОТОРЫЕ МОГУТ СПРОСИТЬ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Модель:         OpenAI GPT (gpt-4o или gpt-3.5-turbo, настраивается через .env)
Эмбеддинги:     OpenAI text-embedding-ada-002 через ChromaDB
Парсинг PDF:    pymupdf4llm (вытаскивает текст с сохранением структуры)
Парсинг DOCX:   python-docx
PDF-отчёт:      WeasyPrint 65.1 + pydyf 0.11.0 (рендерит HTML → PDF)
Стриминг чата:  Server-Sent Events через FastAPI StreamingResponse
Auth:           JWT (python-jose), пароли — pbkdf2_sha256 (passlib)
Миграции БД:    Alembic (2 миграции: initial + analysis_metadata)
Тесты:          pytest (unit-тесты на pipeline, parser, rag_service)

  → «OpenAI API ключ мог истечь, в production используется ваш ключ.
     Покажу архитектуру и уже готовые результаты.»

Если PDF не скачивается:
  → «WeasyPrint иногда медленно рендерит на слабых машинах.

Если контейнер упал:
  → docker compose up -d  (поднять снова)
  → docker compose logs backend --tail=30  (посмотреть ошибку)

Если БД не отвечает:
  → docker compose restart db backend
